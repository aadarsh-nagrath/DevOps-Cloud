## 1. rspec-puppet — Unit Testing

`rspec-puppet` compiles a catalog for a class/define **in memory** (no real system changes) and asserts on the resources it contains — fast feedback without provisioning anything.

### Directory convention
```
mymodule/
├── manifests/
│   └── nginx.pp
└── spec/
    ├── classes/
    │   └── nginx_spec.rb
    └── spec_helper.rb        # generated by PDK, wires up rspec-puppet
```

### Example spec
```ruby
# spec/classes/nginx_spec.rb
require 'spec_helper'

describe 'profile::nginx' do
  # Test against multiple OS facts to catch OS-conditional bugs
  on_supported_os.each do |os, os_facts|
    context "on #{os}" do
      let(:facts) { os_facts }

      it { is_expected.to compile.with_all_deps }   # catalog compiles without error, all deps resolve

      it { is_expected.to contain_package('nginx').with_ensure('installed') }

      it {
        is_expected.to contain_service('nginx')
          .with_ensure('running')
          .with_enable(true)
      }

      it {
        is_expected.to contain_file('/etc/nginx/nginx.conf')
          .that_requires('Package[nginx]')   # assert relationship, not just presence
          .that_notifies('Service[nginx]')
      }

      # Parameterized test
      context 'with worker_processes => 8' do
        let(:params) { { 'worker_processes' => 8 } }
        it { is_expected.to contain_file('/etc/nginx/nginx.conf').with_content(/worker_processes 8;/) }
      end

      # Negative test — assert something is NOT present
      it { is_expected.not_to contain_package('apache2') }
    end
  end
end
```

### Running
```bash
# Via PDK (recommended — manages the Ruby/gem environment for you)
pdk test unit

# Or directly with bundler if not using PDK
bundle exec rspec spec/classes/nginx_spec.rb
```

## 2. puppet-lint & PDK

### `puppet-lint` — style/convention checker
```bash
puppet-lint manifests/init.pp
# WARNING: line has more than 140 characters on line 12
# WARNING: indentation of => is not properly aligned on line 20

# Auto-fix simple issues
puppet-lint --fix manifests/init.pp
```
Common checks: arrow (`=>`) alignment, trailing whitespace, hard tabs, 140-char line length, double-quoted strings without interpolation (should be single-quoted), unquoted resource titles.

### PDK (Puppet Development Kit) — scaffolding, validation, testing in one tool
```bash
# Scaffold a brand-new module with the standard directory layout, gemfile, rubocop/lint config
pdk new module nginx --skip-interview

# Scaffold a new class inside an existing module (creates manifest + matching spec stub)
pdk new class nginx::config

# Scaffold a new defined type
pdk new defined_type nginx::vhost

# Scaffold a task
pdk new task nginx restart

# Validate: runs puppet-lint, puppet parser validate, metadata.json schema check, and Ruby syntax check
pdk validate

# Run only one validator
pdk validate puppet-lint
pdk validate metadata

# Run unit tests (wraps rspec-puppet, manages bundler env automatically)
pdk test unit
```
PDK pins a known-good, consistent Ruby/gem toolchain via its own bundled Ruby — avoids "works on my machine" drift between engineers' local Puppet/Ruby versions, which matters a lot given Puppet's Ruby dependency footprint.

## 3. Litmus / Test Kitchen — Acceptance/Integration Testing

Unit tests (rspec-puppet) only prove the catalog *compiles correctly*; they never actually apply it to a real OS. Acceptance testing provisions real (or containerized) target machines, applies the module, and asserts on actual system state.

### Test Kitchen (older, still common)
```yaml
# kitchen.yml
driver:
  name: docker

provisioner:
  name: puppet_apply
  manifest: site.pp

platforms:
  - name: ubuntu-22.04
  - name: centos-8

suites:
  - name: default
    verifier:
      inspec_tests:
        - test/integration/default   # InSpec assertions run against the provisioned container
```
```bash
kitchen converge   # spin up containers, apply the module
kitchen verify     # run InSpec checks
kitchen destroy    # tear down
```

### Litmus (PDK's modern replacement for Test Kitchen, Bolt-based)
```bash
# Added via PDK scaffolding, uses Bolt under the hood to provision + apply + verify
pdk bundle exec rake 'litmus:provision_list[docker]'   # spin up containers per provision.yaml
pdk bundle exec rake 'litmus:install_agent'
pdk bundle exec rake 'litmus:install_module'
pdk bundle exec rake 'litmus:acceptance:parallel'       # run spec/acceptance/*.rb (serverspec-style) tests
pdk bundle exec rake 'litmus:tear_down'
```
```ruby
# spec/acceptance/nginx_spec.rb — runs against a REAL provisioned target, not in-memory
describe 'profile::nginx' do
  before(:all) { run_shell('puppet apply /tmp/nginx_test.pp') }

  it 'installed nginx' do
    result = run_shell('dpkg -s nginx')
    expect(result.stdout).to match(/Status: install ok installed/)
  end

  it 'is listening on port 80' do
    result = run_shell('curl -sI localhost:80')
    expect(result.exit_code).to eq(0)
  end
end
```
Litmus is the current Puppet-recommended tool (Test Kitchen is legacy but still widely deployed in older codebases).

## 4. Dry-Run / What-If Modes

### `puppet parser validate` — syntax check only, no compilation
```bash
puppet parser validate manifests/init.pp
# Fails fast on syntax errors before you even try to compile a catalog
```

### `puppet apply --noop` — full compile, simulate application, make zero changes
```bash
puppet apply --noop manifests/site.pp
# Notice: /Stage[main]/Nginx/Package[nginx]/ensure: current_value absent, should be present (noop)
# Notice: /Stage[main]/Nginx/Service[nginx]/ensure: current_value stopped, should be running (noop)
# Notice: Class[Nginx]: Would have triggered 'refresh' from 1 event

# Same flag works against a real agent run (compiles catalog from the master, doesn't apply)
puppet agent --test --noop
```
`--noop` is the closest Puppet equivalent to `terraform plan` or `ansible --check` — always run it before rolling out a risky manifest change to production, and it's the standard safety gate in CI pipelines (compile + noop-apply against a canary before merging).

## 5. CI Pipeline Example (GitHub Actions)

```yaml
# .github/workflows/puppet-ci.yml
name: Puppet Module CI

on:
  pull_request:
  push:
    branches: [main]

jobs:
  validate-and-test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        puppet_version: ['7', '8']   # test against multiple supported Puppet major versions
    steps:
      - uses: actions/checkout@v4

      - name: Set up Ruby (PDK bundles its own, but rspec needs a base Ruby for the runner)
        uses: ruby/setup-ruby@v1
        with:
          ruby-version: '3.1'
          bundler-cache: true

      - name: Install PDK
        run: |
          curl -LO https://apt.puppet.com/puppet-tools-release-jammy.deb
          sudo dpkg -i puppet-tools-release-jammy.deb
          sudo apt-get update && sudo apt-get install -y pdk

      - name: Validate (lint, syntax, metadata)
        run: pdk validate

      - name: Run unit tests (rspec-puppet)
        env:
          PUPPET_GEM_VERSION: ${{ matrix.puppet_version }}
        run: pdk test unit

      - name: Acceptance tests (Litmus, containers only — skip on forked PRs without secrets)
        if: github.event.pull_request.head.repo.full_name == github.repository
        run: |
          pdk bundle exec rake 'litmus:provision_list[docker]'
          pdk bundle exec rake 'litmus:install_agent'
          pdk bundle exec rake 'litmus:install_module'
          pdk bundle exec rake 'litmus:acceptance:parallel'
          pdk bundle exec rake 'litmus:tear_down'
```
Typical gating strategy: PR must pass `pdk validate` + `pdk test unit` on all supported Puppet versions before merge; acceptance/Litmus runs can be slower and are often a required check only on `main`/nightly rather than every PR, to keep feedback loops fast.

See also: [advanced-and-production-patterns.md](advanced-and-production-patterns.md) for what happens once code passes CI and reaches real agents (catalog compilation, PuppetDB, scaling).
